SVK is a version control system written in Perl that utilizes SVN. Among the many features SVK adds to SVN is the ability to synchronize SVN repositories while maintaining commit changelog information. This article will describe how to configure SVK to to mirror a local development SVN repository to a public read-only SVN repository such as RubyForge. While SVN provides the ability to synchronize repositories with svnsync in versions 1.4 and higher, the destination repository requires a modification to pre-revprop-change hook which is usually not possible with a public repository. SVK does not need this modification, which makes it better suited for the application that is the subject of this article. Other features of SVK make it worth a look but are not discussed here.

The principal information sources used for this article were the SVK Online Book and Creating a Read-Only Mirror of Your SVN Repository with SVK.

Install SVK

yum can be used to install SVK on Fedora.

yum install perl-SVK

Configure an SVK Depot

The SVK depot can be thought of as the dock for all of the repositories managed by the local SVK installation. Begin by creating a depot with a default configuration.

svk depotmap --init

If the user that issued the command is svk the default configuration will create the depot in the /home/svk/.svk directory. To see what just happened list all local depots.

svk depotmap --list

Depot Path ============================================================ // /home/svk/.svk/local

In this listing // is the default depot name, which will be used to reference repositories mirrored in the depot in all future SVK commands, and the indicated path is the default depot path. It is possible to manage multiple depots by giving them names, see the SVK Online Book for more information.

Create SVK Mirrors of SVN Repositories

After creating the depot the SVN repositories to be managed must be mirrored in the depot. First the SVK mirror of the local project repository will be created and then a RubyForge SVK mirror. When creating the SVK mirrors the full name of the mirror, including the mirror depot, and the URL to the SVN repository must be specified.

svk mirror //has_ancestor_local http://localhost/svn/has_ancestor svk mirror //has_ancestor_rubyforge svn+ssh://user@rubyforge.org/var/svn/has-ancestor

List all mirrors to see what just happened.

svk mirror --list

Path Source ============================================================================= //has_ancestor_local http://localhost/svn/has_ancestor //has_ancestor_rubyforge svn+ssh://user@rubyforge.org/var/svn/has-ancestor

In this listing the items under Path, //has_ancestor_local and //has_ancestor_rubyforge, are the SVK paths to the mirrors, // indicates the containing depot, and the items under Source are the respective URLs of the mirrored SVN repositories. The SVK mirror paths will be used to reference the mirrors in all SVK commands.

The SVK mirrors must now be initialized by synchronizing with their SVN repositories. Since the RubyForge repository is assumed to be read-only this is the only time synchronization needs to be performed.

svk sync //has_ancestor_local svk sync //has_ancestor_rubyforge

Mirror Local SVN Repository to Public SVN Repository

SVK is now configured to mirror the local repository //has_ancestor_local to remote public repository //has_ancestor_rubyforge. The first synchronization must be done manually and subsequent synchronizations may be automated. This is required to cache authentication credentials for http and svn. Caching of credentials is not supported for ssh+svn, which is used by RubyForge, so public key authentication must be used.

Synchronization of the repositories only requires two commands. The first synchronizes //has_ancestor_local and the second merges //has_ancestor_local with //has_ancestor_rubyforge. It should be noted that the SVK command used is smerge not merge.

svk sync //has_ancestor_local svk smerge --baseless --incremental --verbatim //has_ancestor_local //has_ancestor_rubyforge

Automation

The principal challenge in automating synchronization of the local SVN repository to remote SVN mirror is locating the SVK depot and configuration. SVK looks for the depot configuration file in $HOME/.svk/config and also appears to create all depots in $HOME/.svk. Here $HOME is the $USER home directory. So, for any automation mechanism the $USER executing the the SVK commands must have a $HOME/.svk. There are many ways to solve this problem. Here a couple of possible solutions will be discussed.

The simplest method, and one that I use, is to create a user called svk, create the depots in this user account and write a script containing the two SVK commands that perform synchronization. Synchronization can be performed periodically by creating a cron job that executes the synchronization script or by manually executing the script.

If more timely synchronization is required it is possible to modify the local SVN repository post-commit hook to execute the SVK synchronization commands. Creating a Read-Only Mirror of Your SVN Repository with SVK describes how to do this for Debian Linux dialects which access SVN through http. It seems straight forward to extend the solution to Redhat Linux dialects but this only solves the problem for http access. It looks as though the solution could be extended to svn but a separate implementation likely would be required because the $USER for http and svn are most likely to be different. It is not clear to me if same solution could be extended to svn+ssh. Also, I believe one solution that supported all three protocols is likely to require a different approach.

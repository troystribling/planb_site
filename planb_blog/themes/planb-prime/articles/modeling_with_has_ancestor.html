has_ancestor is a Ruby on rails plugin that enables the specification of inheritance relations between ActiveRecord models. Currently in Rails this functionality is only available in Single Table Inheritance. Single Table Inheritance requires that all models in an inheritance hierarchy be in the same table, which is acceptable if the number of models in the hierarchy is small. For inheritance hierarchies that contain a large or unknown number of models more complex representations of inheritance relations, such as Class Table Inheritance, are required. has_ancestor provides and implementation of Class Table Inheritance based on rails Polymorphic Associations. The implementation used here differs from the one used used in the Class Table Inheritance Plugin, which is based on database views.

In this article an example implementation of models using has_ancestor will be presented. In Release 0.0.0 there are some limitations in usage, which are discussed, but that are not prohibitive. An RSpec report for this example is available and the spec implementation is provided with the plugin. See the has_ancestor project page for project information and documentaion.

Model Migration

The example is a simple representation of a line item in a purchase. It consists of three models. The first, item_purchase, is the root of the inheritance hierarchy. It has two descendant models, stock_item_purchase and contract_item_purchase. The migration for the model is shown below. Note that the primary_key for each model is assigned a non-default value. This is done to avoid name collisions between models in the inheritance hierarchy. In general the primary key of a model must be of the form #{model}_id where model is the tablelized from of the model class name.

class CreateHasAncestorModel < ActiveRecord::Migration def self.up create_table :item_purchases, :force => true, \ :primary_key => :item_purchase_id do |t| t.column :item_purchase_descendant_id, :integer t.column :item_purchase_descendant_type, :string t.column :closed, :integer, :default => 0 t.column :item, :string t.column :unit_cost, :integer t.column :unit_count, :integer t.column :cost, :integer end create_table :stock_item_purchases, \ :force => true, \ :primary_key => :stock_item_purchase_id do |t| t.column :in_inventory, :integer, :default => 0 end create_table :contract_item_purchases, \ :force => true, \ :primary_key => :contract_item_purchase_id do |t| t.column :length, :integer end end def self.down drop_table :item_purchases drop_table :stock_item_purchases drop_table :contract_item_purchases end end

Model Classes

In this section the model class definitions are specified. Within the model class definitions ancestor and descendant associations are declared. Below the item_purchase class is defined. Notice the has_ancestor declaration and the definition of the close_item_purchase method. For this model the cost of a purchased item is product of the unit_cost and unit_count.

class ItemPurchase < ActiveRecord::Base has_descendants def close_item_purchase self.cost = self.unit_count * self.unit_cost self.closed = 1 end end

The stock_item_purchase class is defined below. The ancestor association with item_purchase is specified and since no implementation of close_item_purchase is given the ancestor implementation will be used.

class StockItemPurchase < ActiveRecord::Base has_ancestor :named => :item_purchase end

The contract_item_purchase class is defined below. The ancestor association with item_purchase specified but close_item_purchase has a new implementation, namely, the product of the unit_cost, unit_count and contract length.

class ContractItemPurchase < ActiveRecord::Base has_ancestor :named => :item_purchase def close_item_purchase self.cost = self.length * self.unit_count * self.unit_cost self.closed = 1 end end

Use

In this section several examples illustrating use of the model defined in the previous section will be described.

Construction of stock_item_purchase and contract_item_purchase objects is illustrated. Note that attributes of the item_purchase ancestor model are specified in both constructors and in addition to attributes belonging to the model.

stock_item = StockItemPurchase.new(:item=>'keyboard',\ :unit_cost=>300,:unit_count=>1) contract_item = ContractItemPurchase.new(:item=>'CEO',\ :unit_cost=>30000, :unit_count=>1, length=>180)

Model setters and getters work for attributes defined on model and attributes defined on item_purchase.

stock_item.in_inventory = 1 stock_item.unit_count = 2 last_item = stock_item.item contract_item.unit_count = 2 contract_item.length = 240 last_item = contract_item.item

ActiveRecord::Base finders work for attributes defined on model and attributes defined on item_purchase. If a finder is called that filters on multiple attributes all attributes must be on the same model. Also, if the find attribute is an attribute on an ancestor model an instance of the ancestor type will be returned. To return an object of the required type to_descendant must be used. It should be noted that the descendant attributes will be retrieved from the database when to_descendant is called.

stock_item = StockItemPurchase.find_by_item('keyboard').\ to_descendant(:stock_item_purchase) stock_items = StockItemPurchase.find_all_by_in_inventory(1) stock_items = ItemPurchase.find_all_by_unit_cost_and_item(50, \ 'keyboard').to_descendant(:stock_item_purchase) contract_item = ContractItemPurchase.find_by_item('CEO').\ to_descendant(:contract_item_purchase) contract_items = ContractItemPurchase.find_by_length(240)

close_item_purchase called from stock_item_purchase calls the implementation on item_purchase but calls the the model implementation when called from contract_item_purchase.

stock_item.close_item_purchase contract_item.close_item_purchase

A polymorphic like behavior for method calls can be obtained by calling to_descendant with no arguments. It should be noted that the descendant attributes will be retrieved from the database when to_descendant is called.

ItemPurchase.find_by_closed(0).each do |item| item.to_descendant.close_item_purchase item.to_descendant.save end

Limitations

There are some issues with the current implementation that will be addressed in the future that are illustrated in the following examples.

If an ActiveRecord::Base find is called that filters on attributes in different models in the inheritance hierarchy a NoMethodError exception will be raised.

ItemPurchase.find_all_by_unit_cost_and_length(50, 100)

If an ActiveRecord::Base find is called on an ancestor attribute that matches an attribute in some other descendant that descendant will be returned. The example below will return an item_purchase object that has a stock_item_purchase not a contract_item_purchase object as descendant. This would be completely unexpected.

ContractItemPurchase.find_by_item('keyboard')
